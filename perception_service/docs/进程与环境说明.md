# perception_service 进程与环境说明

本文档固定当前 `perception_service` 的运行环境边界，避免把 API 层、`SAM3` 和 `SAM3D-object` 的依赖混装到同一个环境里。

## 1. 总体原则

- `pointmap` 固定在 `perception_service` 内部生成，不由决策层上传。
- API 层环境保持轻量，只负责 HTTP API、artifact 管理、基础预处理和推理调度。
- `SAM3` 与 `SAM3D-object` 继续使用各自独立的 conda 环境运行。
- 不要把 `SAM3` / `SAM3D-object` 的重依赖直接安装到 API 层的 `.venv` 里。

## 2. 进程与环境分工

| 进程 | 作用 | 环境类型 | 环境名 / 路径 |
| --- | --- | --- | --- |
| API 进程 | 提供 `POST /artifacts`、`GET /artifacts/{id}/content`、`POST /perception/infer`，并负责 `depth + intrinsics -> pointmap`、artifact 管理、调度模型子进程 | `uv` 项目虚拟环境 | `/root/robot_task/perception_service/.venv` |
| SAM3 进程 | 负责 2D 分割，生成实例 mask / bbox | conda | `sam3` -> `/root/autodl-tmp/conda/envs/sam3` |
| SAM3D-object 进程 | 负责单目标 3D 重建，消费 RGB、mask 和感知层内部生成的 `pointmap` | conda | `sam3d-objects` -> `/root/autodl-tmp/conda/envs/sam3d-objects` |

## 3. API 层环境

当前 API 层项目位于：

- 项目根：`/root/robot_task/perception_service`
- `uv` 项目文件：`pyproject.toml`
- 锁文件：`uv.lock`
- 虚拟环境：`.venv`

当前本机实际 `.venv` 信息：

- Python：`3.12.3`
- 解释器来源：`/root/miniconda3/bin/python3`
- `uv` 版本：`0.11.2`

这层环境只保留轻量依赖：

- `fastapi`
- `uvicorn`
- `python-multipart`
- `numpy`
- `pillow`

这层环境不负责直接承载：

- `ultralytics` 的完整 `SAM3` 推理依赖
- `SAM3D-object` 的 CUDA / `pytorch3d` / `flash_attn` / `kaolin` 依赖

## 4. SAM3 环境

`SAM3` 相关脚本和适配代码放在：

- `perception_service/sam3-ultralytics/`
- 当前 API 调用的桥接脚本：
  - `perception_service/sam3-ultralytics/run_sam3_inference.py`

但真正执行 2D 分割的运行环境固定为：

- conda 环境名：`sam3`
- 环境路径：`/root/autodl-tmp/conda/envs/sam3`

建议调用方式：

- 优先用环境内解释器绝对路径：
  - `/root/autodl-tmp/conda/envs/sam3/bin/python`
- 或者显式使用：
  - `conda run -n sam3 python ...`

## 5. SAM3D-object 环境

`SAM3D-object` 上游仓库位于：

- `perception_service/SAM3D-object/`
- 当前 API 调用的桥接脚本：
  - `perception_service/backend_scripts/run_sam3d_inference.py`

真正执行 3D 重建的运行环境固定为：

- conda 环境名：`sam3d-objects`
- 环境路径：`/root/autodl-tmp/conda/envs/sam3d-objects`

建议调用方式：

- 优先用环境内解释器绝对路径：
  - `/root/autodl-tmp/conda/envs/sam3d-objects/bin/python`
- 或者显式使用：
  - `conda run -n sam3d-objects python ...`

## 6. pointmap 归属

这里单独写死当前约定：

- `llm_decision_making` 不上传 `pointmap`
- `robot_service` 提供 `RGB`、`depth` 和 `camera_intrinsics`
- `perception_service` 在 API 层内部把 `depth + camera_intrinsics` 转成 `pointmap`
- 内部生成的 `pointmap` 再传给 `SAM3D-object`

也就是说：

- `pointmap` 是感知层内部中间结果
- 不是决策层请求字段
- 不是跨模块上传 artifact

## 7. 当前推荐启动方式

如果后面把三层拆成独立进程，推荐保持下面的责任边界：

1. API 主进程运行在 `perception_service/.venv`
2. API 主进程调 `sam3` 环境里的桥接脚本做 2D 分割
3. API 主进程调 `sam3d-objects` 环境里的桥接脚本做 3D 重建
4. API 主进程统一组装协议响应

这样做的原因是：

- API 层依赖最轻
- 模型依赖互不污染
- 后续替换 `SAM3` 或 `SAM3D-object` 时不会影响 API 环境
